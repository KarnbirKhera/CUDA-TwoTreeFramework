When we write a GPU kernel, two separate sets of decisions are happening at the same time. One is about the algorithm — what mathematical operation is being performed, and how does it work? The other is about the architecture — where does each piece of data live during execution, which threads handle which work, and when does data move between memory levels?

When we often start learning parallel computing, we often focuse entirely on the first set and guessed at the second. That produces kernels that give correct answers but run far slower than the hardware is capable of. The data lives in the wrong places, moves at the wrong times, and gets shared through the wrong channels.

This framework is a structured way to make those architectural decisions on paper before writing any code. It covers kernels where every contracted dimension uses an associative and commutative operator — which turns out to be a wide and important class. Each question builds directly on the answers before it. By the time we open our editor, every structural decision is already made. We're just translating a completed design into code.

The framework has five numbered steps, but Step 1 has a decision point in the middle. We work through Step 1a, make a classification decision, then complete Step 1b. Understanding why the steps are in this order is what makes the framework work on kernels we've never seen before. That's what I've tried to explain here.

Framework scope — the algebraic property that makes derivation mechanical

I've used this framework to derive kernel architectures for operations where every REDUCE dimension uses an associative and commutative operator — specifically vector add, transpose, shared reduced sum, GEMM, dense attention, paged KV MLA, and sparse attention. The operators that fit this pattern include addition, multiplication, max, min, bitwise AND/OR/XOR, dot product, and online softmax merge — any body where partial results can be combined in any order and produce the same final answer.

That algebraic property is what makes the derivation mechanical. Because partial results can be combined in any order, the framework can assign REDUCE dimensions to any level of the hierarchy — warp shuffle, block merge, or sequential thread-private accumulator — and the partial results compose consistently regardless of which threads produced them. The binding table, FSM, lifetime tables, and index trees all follow from this single property. Without it, the derivation method breaks down.

If a REDUCE dimension uses an operator that isn't associative and commutative — subtraction, selection (top-K), scan (prefix sum) — the framework is probably at its edge for that operation. Those tend to require different phase structures and intermediate tensors that the binding table isn't set up to derive. Catching that at Step 1a saves a lot of wasted work further in.

hardware AREA dims cost zero inter-block sync — blocks never communicate. REDUCE dims must touch sync hardware: shuffles, barriers, or atomics to combine partial results.
Step
1a

Dimension Fate

"What does the math do to each dimension — does it survive into the output, or disappear?"

This is always the first thing we do with any tensor operation. We don't think about threads yet, we don't think about memory yet — we look only at the mathematical operation and ask a single precise question about each dimension: does this dimension appear in the output tensor, or does it get consumed by the computation?

To answer this, write down the shape of every input tensor and the shape of every output tensor. Then go through each dimension by name and find it in the output. It's either there, or it isn't.

What counts as an output tensor? Only tensors that are written to global memory at the end of the kernel. If the operation produces an intermediate value that is consumed within the same kernel and never written out — a running max for numerical stability, a partial dot product, a row sum used to normalize — it is an internal intermediate, not an output tensor. Do not list it here. Internal intermediates appear later as state variables in the REDUCE dimension's description, or as temporary smem buffers in Step 2. If we're unsure whether something is an output or an intermediate, ask: "does the caller of this kernel read this value?" If yes, it is an output. If no, it is an intermediate.
Survived dimensions — AREA

A dimension that appears in the output is called an AREA dimension. It means the operation produces one independent result for every position along this dimension. Computing the result at position 3 requires no knowledge of what happened at position 7. These positions are completely independent of each other.

This independence is valuable because it means we can assign different blocks of threads to different positions along an AREA dimension, and those blocks never need to communicate. AREA dimensions are what make parallelism cheap.

Contracted dimensions — REDUCE

A dimension that does not appear in the output is called a REDUCE dimension. It means all the values along that dimension are being combined into a smaller result — summed, compared, selected from, or otherwise collapsed. The dimension gets consumed by the computation.

We can still parallelize along a REDUCE dimension, but the parallelism costs something: the threads that processed different pieces of it must eventually share their partial results and combine them. This combination step is called a reduction, and it requires synchronization. REDUCE dimensions are what make GPU design interesting.

Once we have identified every REDUCE dimension, write down the reduction body for each one in plain language — the specific operation that combines elements along that dimension. Is it addition? A maximum? A comparison-and-swap? A heap insertion? Be specific, because the body operation determines what state variable a thread needs to carry through the computation.

That state variable is the piece of information a thread holds from one element to the next as it processes a REDUCE dimension. For a running sum, the state variable is a single number initialized to zero. For a running maximum, it's a single number initialized to negative infinity. For a heap-based selection, it's the heap itself — an array of k slots, each holding a (value, position) pair. Write down the type of the state variable and how it's initialized, because this will appear directly in later steps.

A common mistake: skipping the state variable description because "it's obvious." It rarely is, and the state variable type is what determines how much register space a thread needs and whether the reduction can be done with warp shuffles or requires shared memory. Write it down explicitly.
Reduction character — the single gate before continuing

Alongside the reduction body and state variable, verify the reduction character of each REDUCE dimension. This framework covers exactly one character:

Parallel — the only character this framework handles. The body operation is associative and commutative. This means any two partial results can be combined in any order and produce the same final answer. This algebraic property is what makes the binding table, FSM, and warp shuffle derivation mechanical: because order of combination doesn't matter, we can freely assign slices of the REDUCE dimension to different threads, warps, and blocks without coordinating which slice each one processes. Warp shuffle reductions, block-level binary trees through shared memory, and sequential thread-private accumulators are all correct implementations of the same associative+commutative reduction — the framework tells us which level to use and how to connect them. Common bodies with this character: addition, max, min, product, bitwise AND/OR/XOR, dot product, online softmax merge (the fmaxf + rescaled sum combination), and any weighted sum.

If a REDUCE dimension is not associative and commutative — for example, selection (top-K, argmax) or scan (prefix sum, running count) — the framework is probably at its edge for that operation. Those use different parallel algorithms (warp min-heaps, radix select, Blelloch scan) that introduce intermediate tensors and phase structures I haven't been able to fit into this derivation. They're worth a different starting point.

Quick character lookup — parallel bodies covered by this framework:

Parallel (proceed): sum · product · max · min · bitwise AND/OR/XOR · count of matching elements · weighted sum · dot product · online softmax merge (max + rescaled sum) · any body where partial results can be combined in any order regardless of which threads produced them.

Not sure? Ask: "if I give two threads half the data each, can they each produce a partial result that I can combine afterward without needing to see the full dataset?" If yes, parallel — proceed. If the merge step requires knowing the full dataset (e.g. the k-th boundary score in selection, or all prior outputs in a scan), that operation is likely outside what this derivation can handle — worth noting before going further.

Step 1

Algorithm Classification

"Which algorithm family implements this reduction, and what does the output need to satisfy?"
Why this fires here — not at the beginning

This decision needs two things as input: the body operation of the REDUCE dimension (what are we combining?), and the relative sizes of the dimensions involved (how much data is being reduced, and how large is the result?). Neither of those is available before Step 1a. We can't classify the algorithm before we know what the algorithm is doing.

This decision also needs to happen before Step 1b, because the algorithm we choose determines which tensors exist in your kernel. A multi-pass algorithm has intermediate tensors that a single-pass algorithm doesn't. Analyzing access patterns before knowing which algorithm we're using means you might analyze the wrong tensors entirely.

Look at the REDUCE dimension and its body operation — which we have already verified is associative and commutative. Ask yourself: is there a known algorithm family that implements this body efficiently on a GPU? Most parallel reductions fall into one of a small number of families. Simple reductions — sum, min, max — use a parallel reduction tree. Matrix multiplications use tiled GEMM. Attention uses fused online softmax. Each family has its own well-studied structure that determines what intermediate tensors exist and how the FSM phases are ordered.

For each candidate family, consider the relevant problem dimensions. If there are multiple family options, the right one is usually determined by a ratio or threshold — the size of the output relative to the size of the input, the sparsity of the data, or a threshold below which one approach dominates and above which another does. Write down the ratio, compute it with your actual dimension sizes, and commit to a specific family. Then name it explicitly at the top of your derivation.

Not recognizing an algorithm family for the operation is a useful signal — it's happened to me several times during this competition, and it usually means the algorithmic landscape for that problem is worth understanding before the derivation starts. The framework takes over once a family is chosen, but the choice itself has to come from somewhere outside it.

Common algorithm families — parallel reductions only. All entries below have associative and commutative reduction bodies. Match your reduction body to the closest entry, then use the listed family name in your derivation record. Operations outside this list (prefix scan, selection/top-K, sort) are outside this framework's scope.

Parallel reduction (sum / max / min / AND / OR): Family = parallel reduction tree. Warp-level: shuffle-down. Block-level: binary tree through smem. No intermediate tensors.

Matrix multiply (C = A × B, reduction over inner dimension K): Family = tiled GEMM. Tiles of A and B staged in smem; each thread accumulates a private output tile. No intermediate tensors unless split-K is used.

Attention (softmax-weighted sum of V, reduction over sequence length J and head dim K): Family = Flash Attention (fused online softmax). Intermediate tensors: running state (m, l, o) in registers per query; per-tile scores in smem. No S[N,N] matrix. The online softmax merge — fmaxf + rescaled sum — is parallel character.

Softmax (normalize a vector, reduction over the vector dimension): Family = two-pass (max then exp+sum) or online single-pass. Intermediate tensors: row max and row sum if two-pass; online state (m, l) if single-pass.

Layer norm / RMS norm (normalize across hidden dim): Family = two-pass reduction (mean/variance then normalize). Intermediate tensors: mean, variance per row.

Histogram (count occurrences, reduction over input positions): Family = privatized histogram (one histogram copy per warp or block, then merge). Warp primitive: match (__match_any_sync) to avoid serialization at bin collisions.

Convolution (sum over kernel spatial positions and input channels): Family = direct tiled convolution (like GEMM but with filter reuse) or implicit GEMM (reshape to matrix multiply). Choose based on whether input padding is needed and filter size.

Output contract. Before moving on, also write down what properties the output must satisfy beyond just having the right shape. These constraints are part of the problem specification but are easily forgotten until they surface as bugs late in development. Does the output need to be sorted in a particular order? Do values need to sum to one, indicating renormalization is required? Is the output a single randomly sampled index rather than a set of values? Does the operation need to be numerically stable or deterministic across runs? Whatever the answer, write it down — then immediately name the FSM phase each constraint generates. A renormalization requirement becomes a NORMALIZE phase. A sort requirement becomes a SORT phase before STORE. A numerical-stability contract becomes a running-max initialization in INIT. Recording the contract and the phase it produces together prevents the class of bug where a contract requirement is written down in Algorithm Classification and then forgotten before Step 3.

1b

Access Patterns

"How is each tensor actually read or written — what kind of indexing does each dimension use?"

With the algorithm chosen, we can complete the geometry step by examining how each tensor is accessed. For every dimension of every tensor — inputs, outputs, and any intermediate tensors your chosen algorithm introduces — we assign a leaf type that describes how the index into that dimension is computed. This classification matters because different leaf types have different performance characteristics and generate different code.

Two categories of leaves. Every index expression contains two kinds of leaves: execution leaves that describe how an address is computed from thread and loop variables, and value leaves that describe what happens to the data once we reach it. Classify both for every access.
— Execution leaves —
Describe how the physical address is computed from execution context.
Arithmetic
The address is a linear combination of execution variables: base + i₁·s₁ + i₂·s₂ + … using only addition and multiplication. The default and best case — the hardware can predict these accesses and adjacent threads often access adjacent memory (coalescing). Example: A[row * K + col] where both row and col are computed from thread/block indices.
Composite
A 1D flat index is unpacked into 2D (or higher) coordinates using integer division and modulo: row = idx / N, col = idx % N. Used when threads are linearized (e.g. tid = ty·blockDim.x + tx) but smem is indexed in 2D. Division and modulo cost extra instructions but do not add memory round-trips. This is the Flatten→Stride→Unflatten pattern used in cooperative smem loads: flatten thread coordinates into a single tid, stride it across the buffer, then unflatten back to 2D smem indices. Concrete example: for a tile A_smem[BK][BM] and thread index tid, the mapping is row = tid / BM (K dimension, 0..BK-1) and col = tid % BM (M dimension, 0..BM-1), giving A_smem[row][col] = A[global_row * K + global_col].
Lookup
The physical address cannot be computed directly — we must first read an index array to obtain it: global_array[ index_array[arithmetic_expr] ]. Lookup leaves add a full extra memory round-trip (~200 cycles) before the real data access and prevent hardware prefetching. Record them explicitly — each one surfaces as a design concern in Step 2 and as an indirection row in the index expression table. Example: K[ kv_indices[row_ptr[q] + tile] · T_J + j ] in sparse attention.
— Value leaves —
Describe what happens to the value at the computed address.
Type conversion
The data is stored in one numeric type but computed in another. A conversion instruction executes on every element access. Common in neural network kernels where weights are stored in FP8 or FP16 but computations occur in float32. When we identify a type-conversion leaf, note the storage type, the computation type, and the conversion — it will appear explicitly in the COMPUTE phase of the FSM. Example: __half2float(A_fp16[row * K + col]).
Conditional
A runtime condition determines whether the access happens at all: condition ? value : default. Two distinct placements follow — the placement is determined by the triggering iterator, not by the condition itself.

Grid-level conditional → early exit (Bounds Check phase). If the triggering iterator is a grid-level dimension (blockIdx), place a cheap early-exit guard at the start of the kernel before any shared memory is touched.

Tile-level conditional → predicated load (inside the tile loop). If the triggering iterator is a tile loop variable at Block or Warp granularity, the response is a predicated load or store inside the loop — never an early exit. An early exit here would leave other threads waiting at a subsequent __syncthreads() that is never reached, causing a deadlock.
Compound index expressions. An index can contain multiple leaf types in the same expression — a lookup leaf wrapping composite and arithmetic leaves is common in paged systems. When this happens, classify the entire access by the highest-cost execution leaf present, using this priority: Lookup > Conditional > Type conversion > Composite > Arithmetic. A single lookup leaf makes the whole access a lookup-access regardless of the arithmetic around it. Record the lookup leaf explicitly because each one adds a full extra memory round-trip before the real data is reached.

Work through every tensor systematically and don't assume any dimension is arithmetic without checking. A dimension that appears simple at first might actually require a lookup, and missing that will mean missing a significant design concern before we've written any code. By the end of Step 1b, we have a complete picture of what your kernel needs to compute — the dimension fates, the chosen algorithm, the output contract, and the access structure of every tensor involved.

The Binding Table

"Where in the GPU's execution hierarchy does each dimension live, and who is responsible for it?"

This is the most important design step in the framework. Everything up to this point was about understanding the problem. Step 2 is about designing the solution — specifically, assigning each dimension of your operation to a level of the GPU's execution hierarchy.

To fill in the Binding Table, we first need to understand what that hierarchy actually is. Think of it like a company with a very specific org chart, where each level has its own workspace that only people at that level can access.

Grid
Global mem
The whole GPU working on your problem. All blocks can read and write global memory, but it is large and slow — like a warehouse across town. Global memory is where inputs come from and where final outputs go. Blocks in the grid never communicate directly with each other during a kernel; they only coordinate through global memory before or after execution.
Block
Shared mem
A group of threads that share a fast private whiteboard — shared memory. It's much faster than global memory but small (typically 48–164 KB) and visible only within one block. When threads in the same block need to share intermediate results, shared memory is how they do it. Reading from shared memory after another thread wrote to it requires a __syncthreads() barrier.
Warp
Warp prims
Groups of exactly 32 threads that execute in hardware lockstep — when one thread executes an instruction, all 32 do simultaneously. This lockstep enables a family of collective operations — shuffles, votes, matches, and matrix ops — that work register-to-register with no shared memory and no barrier. Reductions are just one member of this family; the warp is a general collective execution unit.
Thread
Registers
Each individual worker with a private notepad — registers. Registers are the fastest memory on the chip and are completely private. State variables, loop accumulators, and intermediate values that belong to one thread's computation live here. The state variable you identified in Step 1a lives in registers during the REDUCE phase.
What tiling means, and when a dimension supports it. Tiling means processing a large dimension in fixed-size chunks rather than all at once, so that each chunk fits in a faster memory level. A dimension supports tiling when processing one chunk either produces a complete independent result, or produces a partial result that combines correctly with partial results from other chunks. For a REDUCE dimension like K in GEMM, each tile of K contributes a partial dot product to the accumulator — the partial results sum correctly, so K is tileable. For an AREA dimension like M, each tile of M produces independent output rows that don't interact at all — trivially tileable. A dimension is not tileable when its chunks cannot be processed independently and cannot produce combinable partial results — for example, a scan dimension where each output depends on all prior outputs.

With granularities recorded, the Binding Table has four columns — one per level. The rows are your dimensions. For each (dimension, column) cell, you answer a specific question about what happens at that level's granularity. Work left to right across the columns for each dimension, because each column's answer often depends on the one to its left.

Grid → Global
Does a block permanently own a fixed slice of this dimension for the entire kernel's execution? For AREA dimensions: usually yes. Assign blockIdx.x or blockIdx.y to this dimension. One block, one slice, owned forever. For REDUCE dimensions: usually no — the whole REDUCE dimension is processed within a single block.
Block → Shared
Would multiple threads in the same block independently load the same data if we didn't stage it? If yes: load that data once into shared memory. All threads read from the whiteboard instead of each making their own slow global memory trip. If no — if every thread reads unique data — staging is wasteful. The key word is "same."
Warp → Primitives
What collective operation does the warp need to perform, and which primitive family implements it? This is not only about reductions. Ask: are threads exchanging or permuting data (shuffles)? Making a collective yes/no decision (votes)? Finding lanes that share a value (match)? Multiplying matrix tiles (tensor cores)? Each has its own primitive family — all faster than shared memory and requiring no barrier.

Inter-warp collective trigger. After selecting the primitive family, compute NUM_WARPS = (blockDim.x + 31) / 32 (ceiling division — always use ceiling, never floor) for every REDUCE dimension that uses a warp primitive. If NUM_WARPS > 1, the warp primitive alone is insufficient — each warp produces one partial result, and those partials must be combined across warps. This requires a shared memory merge buffer: add warp_partial[NUM_WARPS] (or warp_partial[NUM_WARPS][tile_size] for tiled reductions) to the Block→Shared column, and note that two additional __syncthreads() barriers will appear in the FSM — one after each warp's lane 0 writes its partial, one after thread 0 writes the final result. Include this buffer in the smem budget check below.

Compile-time size requirement. Shared memory array dimensions must be compile-time constants in CUDA. NUM_WARPS must therefore be declared as a constexpr or derived from a template parameter or #define — it cannot be a runtime int computed inside the kernel. The canonical pattern is constexpr int NUM_WARPS = (D_HEAD + 31) / 32; placed before the kernel, where D_HEAD is itself a compile-time constant. A static_assert(D_HEAD % 32 == 0, "..."); alongside it documents the assumption that blocks are always full warps.

Active mask rule. Every warp primitive call requires an active mask as its first argument. Never hardcode 0xffffffff — this assumes all 32 lanes are active, which fails for blocks where blockDim.x is not a multiple of 32. Instead, compute the mask explicitly: unsigned mask = __ballot_sync(0xffffffff, true); This returns the bitmask of all active threads participating in the ballot; when subsequently passed to __shfl_down_sync, only bits within the caller's warp are examined by hardware, so threads from other warps have no effect. Some developers prefer unsigned mask = __activemask(); for clarity — both are correct here. The important invariant is that the mask passed to the shuffle matches the lanes that will participate in the reduction.
Thread → Registers
What does a single thread compute entirely on its own? State variables identified in Step 1a live here. One thread's running sum, one thread's current maximum, one thread's heap. Private computation that no other thread needs to see during the COMPUTE phase.

Starting heuristic for thread tile size. A thread tile of 1 (one output element per thread) is always correct and a safe place to start. The only reason to increase it is if Step 4b will show that a register-held value is reused across the tile dimension — meaning we read it from shared memory once but use it multiple times before discarding it. If we're unsure, write 1 here and wait for Step 4b to tell us whether a larger tile is justified. The register budget check below is your ceiling: if the tile-1 design already uses 80+ registers, there is little room to grow.
Warp primitive quick-match. If we're unsure which primitive family applies, match your computation to one of these patterns:

"I need to sum / max / min values across all 32 lanes for the same output element." → Data Exchange (Shuffles), specifically __shfl_down_sync with offsets 16, 8, 4, 2, 1.

"I need all lanes / any lane / how many lanes satisfy a condition." → Voting (__all_sync, __any_sync, __ballot_sync).

"Multiple lanes might write to the same output bucket — I want to elect one representative per group to avoid serialized atomics." → Match (__match_any_sync).

"My REDUCE dimension is a matrix multiply over tiles of shape 16×16×16 (or 16×8×16) and I want hardware tensor cores." → Warp Matrix Ops (wmma::mma_sync). Note that tensor cores fix the tile shape — this constrains your Block granularity in the Binding Table.

"My REDUCE dimension is computed entirely within each thread's private registers and no lanes need to share partial results." → No warp primitive. GEMM with a register tile accumulator is the canonical example: each thread accumulates its own private C tile and never needs to exchange values with other lanes.

After filling in all four columns, do a shared memory budget check. Add up the sizes of everything we've marked as staged in shared memory. For each staged tensor or buffer, multiply the number of elements by the bytes per element. Sum all of it, including any warp_partial buffers generated by the inter-warp collective trigger above. If the total exceeds your hardware's shared memory limit per block (check your GPU's spec sheet — it varies), we have a problem that needs to be resolved now by adjusting tile sizes or reconsidering staging decisions. Finding this on paper is much better than finding it when your kernel silently fails to launch.

Immediately after, do a register budget check. Sum the sizes of all Thread→Registers state variables in float-equivalents. This is a lower bound — the compiler will add temporaries on top. If the sum exceeds 64 floats per thread, occupancy may fall below 4 warps per SM; if it exceeds 128, register spill into local memory (global memory with extra addressing) is likely. Both effects can cost more than the register tiling gains. Resolve by reducing thread-tile shape before proceeding to Step 4a.

Load coverage rule — verify before Step 3. For each shared memory buffer of size S elements, with T active threads in the block: each thread must load ceil(S / T) elements in the LOAD phase. One element per thread is only correct when S = T. The general pattern is a stride loop:

for (int p = threadIdx_linear; p < S; p += blockDim_linear) {
    smem[local_index(p)] = global[global_index(p)];
}

where threadIdx_linear = ty·blockDim.x + tx and the index formulas come from the Composite leaf pattern (Flatten→Stride→Unflatten). If your LOAD phase only loads one element per thread and S > T, it is a correctness bug: part of the smem buffer will contain uninitialized data. Verify the coverage equation for every staged buffer now, before placing barriers in Step 3.

Zero-length reduction guard. After computing num_tiles for any REDUCE dimension, add a guard: if num_tiles == 0, write the identity value and return immediately. Data Exchange (Shuffles)
__shfl_sync __shfl_up_sync __shfl_down_sync __shfl_xor_sync
Any thread needs a value from another lane's register — permutations, reductions, prefix operations, bitonic sort.

The shuffle family is the most general warp primitive. Each instruction lets a lane read the register value of any other lane directly, with no memory and no barrier. The four variants cover different communication patterns.

__shfl_sync(mask, val, src_lane) is a broadcast or gather: every lane specifies which lane it wants to receive from. Different lanes can pull from different sources simultaneously, making arbitrary within-warp permutations possible in a single instruction.

__shfl_down_sync(mask, val, delta) shifts data downward: lane N receives lane N+delta's value. This is the primitive most commonly seen in reductions — you shift by 16, then 8, then 4, then 2, then 1, and lane 0 accumulates the total.

__shfl_up_sync(mask, val, delta) shifts data upward: lane N receives lane N−delta's value. This is the natural primitive for prefix (scan) operations — each lane combines with its left neighbor.

__shfl_xor_sync(mask, val, lane_mask) exchanges between lanes whose IDs differ by a XOR pattern. Butterfly communication networks — which underlie bitonic sort, Walsh-Hadamard transforms, and many parallel algorithms — are wired exactly this way. Every compare-and-swap round of a bitonic sort within a warp is one __shfl_xor_sync call.

Use the shuffle family when threads need data from other lanes. Use __shfl_down for reduction, __shfl_up for prefix scans, __shfl_xor for sort networks and butterfly patterns, and __shfl_sync for arbitrary permutations.

Voting
__ballot_sync __all_sync __any_sync
The warp needs to make a collective decision about its own state — early termination, counting active lanes, finding which lanes satisfy a condition.

The voting family is about collective decision-making rather than data movement. These primitives ask a yes/no question across all 32 lanes simultaneously and return information about how the lanes answered.

__ballot_sync(mask, predicate) returns a 32-bit integer where bit N is 1 if lane N's predicate was true. This gives you the complete picture of which lanes satisfy a condition in one instruction. From the ballot result we can count active lanes with __popc(), find the first active lane with __ffs(), or use the bitmask to coordinate work distribution.

__all_sync(mask, predicate) returns 1 only if every active lane's predicate is true — useful for detecting convergence. If all threads have finished iterating, exit the loop now rather than having each thread check independently and potentially causing warp divergence.

__any_sync(mask, predicate) returns 1 if at least one active lane's predicate is true — useful for detecting whether any thread still has work to do before deciding to continue a loop.

Use the voting family when the warp needs to answer a structural question about itself: are all threads done? How many have valid data? Which ones hit a boundary condition? These questions are expensive when answered per-thread with branches; they are free when answered collectively with a vote.

Match
__match_any_sync __match_all_sync
Multiple lanes need to update the same output position — histograms, sparse accumulations, gather/scatter where threads collide on the same destination.

The match family is the least commonly known but addresses a specific pain point that has no clean solution with the other primitives: what do we do when multiple threads in the same warp need to update the same output element?

The naive answer is atomic operations — but atomics serialize. If eight threads all need to add to the same output bin, those eight additions happen one after another rather than in parallel. The match family lets we find the groups of threads that share a destination and elect one representative from each group to perform the update once.

__match_any_sync(mask, val) takes a value from every lane and returns, for each lane, a bitmask of all the lanes that hold the same value. Each lane learns which other lanes are in its group in one instruction. The group can then use __ballot_sync or __ffs() to elect a representative, who performs the update while the others skip it. The result is the same as serialized atomics but with much less contention.

__match_all_sync(mask, val, pred) checks whether all active lanes hold the same value and returns 1 if so. Less commonly used, but useful for verifying uniform conditions before taking a fast path that assumes uniformity.

Use the match family for histogram operations, sparse scatter/gather, and any situation where threads in the same warp may collide on the same output address.

Warp Matrix Ops
wmma::load_matrix_sync wmma::mma_sync wmma::store_matrix_sync
The REDUCE dimension is a matrix multiply-accumulate and the tile sizes match Tensor Core dimensions (16×16×16 or similar).

The warp matrix family is the most powerful and most specialized. Tensor Cores are dedicated hardware units that perform small matrix multiplications in mixed precision as a single warp-level operation. The throughput gap versus scalar instructions is enormous — hundreds of FLOPs per clock versus a handful.

The mental model here is different from the other families. Instead of each lane holding an independent value that it contributes to a collective, the entire warp collectively owns a matrix fragment distributed across the 32 lanes in a hardware-defined layout. We don't control which lane holds which matrix element — the hardware decides that and optimizes for the tensor core's internal memory structure. You interact through abstract fragment types: a matrix_a fragment, a matrix_b fragment, and an accumulator fragment.

wmma::load_matrix_sync loads a matrix tile from shared or global memory into the warp-distributed fragment representation. wmma::mma_sync performs the matrix multiply-accumulate in one warp-level instruction. wmma::store_matrix_sync writes the accumulator fragment back to memory.

This family changes the Binding Table analysis significantly for matrix-shaped operations. When Tensor Cores are applicable, the Warp column entry is not "use shuffles to reduce" — it's "use wmma fragments to collectively own a tile." The warp is no longer processing 32 independent scalars that communicate; it is collectively processing a matrix as a single unit.

Use this family when your REDUCE dimension is a dot product or matrix multiply, the tile dimensions match the hardware's supported sizes, and the precision requirements are compatible with Tensor Core's mixed-precision modes.

Lifetime Table — Block Level

"Does the reuse count for each Block→Shared buffer actually justify staging it? Validate before building the FSM."

Step 4a is a validation step that runs before the FSM is constructed. The Binding Table just made staging decisions — it said certain data should live in shared memory because multiple threads benefit from sharing it. Step 4a gives you a mechanical way to verify each of those decisions before you build barrier placements around them. A wrong staging decision discovered here costs one revision to Step 2. The same mistake discovered after Step 3 costs a revision to Step 2 and a complete redo of all barrier positions in the FSM.

The underlying principle: staging data into faster memory is only worthwhile if that data is reused enough times to justify the staging cost. If a value is written to shared memory and read exactly once, you paid the staging cost for zero benefit. If it is read a hundred times, you replaced a hundred slow global memory accesses with one global read and a hundred fast shared memory reads. The reuse count determines which case we are in.

How to compute the reuse count for any staged value: Identify which loop iterators are active — still running — while this value is held in shared memory. Then identify which of those active iterators are absent from the value's own index expression. The absent iterators are the ones the value is reused across. Their combined sizes give the reuse count. A reuse count of 1 means staging provides no benefit; a reuse count of the block tile size or larger strongly justifies staging.

Work through every buffer in the Block→Shared column. For each one, compute the reuse count using the rule above. Then apply the check in both directions:

If we staged something in shared memory but its reuse count is 1 or very small, remove it from Block→Shared and revise Step 2 before continuing. The staging is consuming shared memory capacity and memory bandwidth for no gain — and it is reducing the number of blocks that can run concurrently on the SM, which costs occupancy.

If we left something in global memory but its reuse count is large, add it to Block→Shared and revise Step 2. We have missed a staging opportunity that would substantially improve performance.

What counts as "large enough" to justify staging? A reuse count of 1 never justifies staging — you pay the smem write cost for exactly zero reads saved. A reuse count equal to the block tile size or larger (e.g. reuse = BN = 64 for a GEMM A tile) strongly justifies staging. For reuse counts in between, apply this rule: if the smem budget is not yet tight and global memory bandwidth is likely your bottleneck (common for memory-bound kernels), stage it. If the smem budget is already near the limit and the kernel is compute-bound, leave it in global memory. When genuinely unsure, stage it — the default behavior of hardware L2 cache provides a reuse benefit anyway, and explicit smem staging makes that benefit reliable and measurable.

If any revision is made to Step 2 as a result of this check, redo the shared memory and register budget checks before proceeding. Only when the Block→Shared column is validated should we move on to Step 3 to place barriers around it.

The barrier rule — one sentence

Place a __syncthreads() barrier at every transition where thread A has written data into shared memory that thread B in the same block needs to read. If only one thread reads what it wrote, no barrier. If threads in different warps share the data, a barrier is required. The lane binding rule — one declaration per phase

For every phase that uses a warp primitive, write one sentence in your FSM: lane_id indexes dimension ___. This declaration specifies which problem dimension varies across the 32 lanes of the warp during this phase. Every other dimension is handled by a loop inside each lane. Writing it down is the entire rule — the value of doing so comes from two cross-checks it enables.

Cross-check one: consistency with the Compute phase. Compare the lane binding in your Compute phase with the lane binding in your Warp Reduce phase. If they differ, the shuffle is mixing partial results from different output elements rather than combining partial results for the same output element. This is the canonical form of the shuffle corruption bug — lanes compute complete dot products for different tokens, then the shuffle sums those independent dot products together as if they were partial sums for one dot product. The mismatch on paper is the signal. If they must differ (because the kernel genuinely changes what lanes represent between phases), we need a warp-level shuffle for data exchange before the reduction, not a reduction shuffle directly.

Cross-check two: consistency with the reduction character. Look at the dimension your lane binding declares in the Warp Reduce phase. Find that dimension in Step 1a. Since all REDUCE dimensions in this framework are parallel character, a shuffle reduction is valid — provided the lane binding is correct. If the cross-check fails here, the issue is with the lane binding assignment, not the reduction character.

The thread phase-role rule — one declaration per non-warp phase

The dim-binding column must also be filled for phases that do not use a warp primitive. Write: threadIdx indexes dimension ___. This captures the fact that the same thread index can serve different dimensional roles in different phases. In a kernel where threadIdx.x acts as a key-dimension index during a score-computation phase and an output-dimension index during an accumulation phase, the two phases have different thread roles — and unless both are declared, the mismatch is invisible on paper.

Cross-check three: consistency across shared memory boundaries. After declaring the thread role for every phase, find every pair of adjacent phases (A writes smem, B reads smem) and check that the index interpretation of the shared memory buffer is the same in A as in B. If phase A writes smem[threadIdx.x] with threadIdx.x meaning dimension k, and phase B reads smem[threadIdx.x] with threadIdx.x meaning dimension v, the buffer is populated in k-order and consumed in v-order — a silent indexing corruption. The fix is either to reindex the smem read in B, or to recognize that the two phases require a buffer transpose between them. This cross-check catches the bug on paper; without declared thread roles it can only be caught by inspecting a failing numerical output.

Cross-check 3 — a concrete example. Most simple kernels (vector add, reduced sum, GEMM) have a single consistent thread role throughout and will pass this check automatically. This example matters most when a kernel's phases assign threadIdx to different problem dimensions — most commonly in attention kernels where the head dimension is used for both dot-product computation and output accumulation. Suppose a kernel has two phases:

Phase A (LOAD): smem[threadIdx.x] = K_global[j * D + threadIdx.x]. Here threadIdx.x is being used as the head-dimension index k. A 64-thread block fills smem[0..63] with K[j, 0..63].

Phase B (COMPUTE): the code then reads smem[threadIdx.x] while computing a dot product. threadIdx.x still means k. Same role. Cross-check passes.

Now suppose Phase C (ACCUMULATE) also reads smem[threadIdx.x], but this time threadIdx.x is the output-dimension index v. The developer intends to read "my V column." But smem was filled in k-order, not v-order. If k and v both happen to have size D = 64, the read is physically correct (smem[5] when threadIdx.x=5 means the same byte whether k=5 or v=5). Cross-check passes only because D_k = D_v.

If D_k ≠ D_v — for example, in Multi-head Latent Attention where D_k = 128 and D_v = 64 — the same code reads smem[0..63] as v-indices, but smem was filled for k-indices 0..127. Thread 63 reads smem[63] intending V[j, 63], but smem[63] holds K[j, 63], which is only half the K vector. Cross-check 3 would have flagged this: the thread role switches from k-domain to v-domain at the smem boundary, and D_k ≠ D_v makes the physical addresses incompatible.

For each phase, record two things: which state variables are alive — that is, which values are being held and not yet consumed — and the lane binding for any phase that uses a warp primitive. A state variable is alive from its initialization through every phase that updates or reads it, until the phase that finally writes it out to its destination. Tracking this prevents a class of bugs where a state variable is overwritten before its value is used, or consumed in the wrong phase. The lane binding prevents the class of bugs where correctly structured state variables are combined in the wrong way.

Pipelining & Double Buffering

"My kernel is correct — but the memory unit sits idle while threads compute. How do I overlap the two?"

The FSM we built in Step 3 follows a safe, sequential rhythm: load a tile into shared memory → barrier → compute on that tile → repeat. During the compute phase, threads are busy doing arithmetic — but look at what else is happening, or rather, what isn't: the memory unit is completely idle. The GPU's ability to fetch data from global memory sits unused while threads crunch numbers. Then when compute ends and the next load begins, the arithmetic units sit idle while memory is busy. The two pipelines take turns rather than running simultaneously.

Pipelining is the technique that closes this gap. Instead of waiting for tile t's computation to finish before starting tile t+1's load, you fire off the load of tile t+1 in the background while computing on tile t. By the time the computation on tile t finishes, the load of tile t+1 is already done — or at least well underway. Memory and arithmetic run simultaneously rather than alternating.

This sounds simple, but it creates an immediate problem: we can't load tile t+1 into the same shared memory buffer we're currently computing on. Overwriting the buffer mid-computation would corrupt the results. The solution is to allocate two shared memory buffers that alternate roles — a "ping" and a "pong." While we compute on ping, we load into pong. While we compute on pong, we load into ping. This pattern is called double buffering, and it's the only structural change to shared memory that pipelining requires.

The second ingredient is asynchronous copy, available on sm_80 and later via the cp.async instruction. A normal load blocks all threads in the block until the data arrives — that's what makes __syncthreads() after a load necessary. An async copy fires off the memory request and immediately lets threads continue without waiting. Think of it as handing the memory unit a shopping list and saying "get these while I start cooking." Threads proceed to the compute phase on the current tile, and only pause to wait for the async load when they actually need the freshly loaded tile at the top of the next iteration.

Is pipelining worth the added complexity? Only when the compute phase is long enough to cover the load latency. If your load takes 200 cycles and your compute takes 20, the async load finishes long before compute ends — you'll see no benefit. If compute takes 500 cycles and load takes 200, pipelining can almost completely hide the load cost. The Lifetime Table (Step 4) gives you a signal before we write a single line: a high reuse count means compute time dominates, which means pipelining will help. A reuse count of 1 or 2 usually means compute is too short to care.

Lifetime Table — Thread Level

"How many times is each register-held value reused, and does that justify the thread tile chosen in Step 2?"

Step 4b validates the Thread→Registers column of the Binding Table using the same reuse-count method applied to blocks in Step 4a, now applied one level down. The block-level staging decisions were already validated and locked in before the FSM was built. Step 4b now validates the thread-level tiling decisions — specifically, whether the thread tile shape chosen in Step 2's "free" column is actually earning its register cost.

The reuse count at the thread level works identically to the block level: identify which of the thread's owned AREA dimensions are absent from a given register-held value's index. Multiply their per-thread sizes. That product is the register-level reuse count — the number of times one shared memory read is reused across the thread's register tile before the value is discarded.

Thread-level reuse sub-table. For each register-held value, the active iterators are the thread's own loop counters over its owned tile dimensions plus any remaining REDUCE loops. The absent iterators among these are the reuse dimensions at the thread level. A concrete example: suppose a thread owns TM outputs along M and TN along N. It holds a loaded A element indexed by M and K — N is absent. As the thread iterates over its TN owned N outputs, that A element is reused TN times from registers without touching shared memory again. Register-level reuse count for A is TN. Similarly, a B element indexed by K and N has M absent, giving a register-level reuse count of TM.

The practical check: if the Binding Table's Thread→Registers column recorded a thread tile of granularity greater than 1 for any AREA dimension, verify here that at least one value held in registers has a non-trivial reuse count across that tile dimension. If no value does, the larger thread tile is consuming extra registers without benefit — reduce the granularity back toward 1 and recheck the register budget from Step 2. If a value does have a large reuse count but the thread tile was left at 1, we have missed a register tiling opportunity that would reduce shared memory traffic substantially — increase the thread tile shape and recheck the register budget.

The Binary Tree Table (Index Trees)

"How is each memory address computed, and what does the structure of that computation reveal?"

The final step is the most mechanical one, and it serves two distinct purposes: verification and opportunity detection. Every time your kernel reads from or writes to memory, it computes an address. That address is always an expression built from execution indices and loop variables. Step 5 asks you to write each of those expressions as a tree, where the leaves are labeled with their type from Step 1b.

The tree doesn't need to be elaborate. The root is the full address. The branches split it into components. The leaves are the individual terms — each one labeled as arithmetic, lookup, type-conversion, or conditional. A simple access like data[batch * N + i] produces a tree with two arithmetic leaves: one for batch × N and one for i × 1. A paged access adds a lookup leaf at the top before the arithmetic leaves within the page.

Verification. Once we have the tree, computing the maximum possible address is straightforward: substitute the maximum value for each leaf (for a dimension of size N, the maximum index is N−1) and evaluate the expression. Check that this maximum address falls within the allocated size of the corresponding buffer. Do this for every access in your kernel. It's tedious but mechanical, and it catches buffer overflow bugs before they happen — bugs that are notoriously difficult to track down once the code is running on a GPU.

Opportunity detection — upward pass. Look at your REDUCE dimensions — the ones that are contracted — and find them in the index trees. For each REDUCE dimension that appears as an arithmetic leaf, ask: at what level of the execution hierarchy is this dimension currently being processed? Then ask the follow-up question: what would happen if it were processed at the next level up?

If a REDUCE dimension currently owned by individual threads appears as an arithmetic leaf with a simple stride, moving it to the warp level would mean all 32 threads in a warp access 32 consecutive memory positions simultaneously. That's perfectly coalesced global memory access — one of the most significant optimizations available on a GPU. If a REDUCE dimension currently processed sequentially within a thread has no data dependencies between its iterations, it could be parallelized across warp lanes and reduced with shuffle instructions.

Opportunity detection — downward pass. Look at your AREA dimensions and find them in the index trees. For each AREA dimension that appears as an arithmetic leaf at Block or Warp granularity, ask: what would happen if a slice of it were moved down into the thread's register tile? If threadIdx currently addresses a single element of this dimension, and the kernel iterates over a REDUCE dimension while holding that output element in a register, then the output element is being reused exactly once per REDUCE step — which is the minimum. Increasing the thread tile to T elements along this AREA dimension would mean one shared memory read is reused T times across the tile, replacing T smem reads with T register reads for each loaded value. If the Step 4b reuse table already recorded a large thread-level reuse count for this dimension, a tile of T=1 is wasteful — the downward analysis makes the missed opportunity explicit. Record the recommended thread tile size and note that Step 2's thread-tile entry should be revisited before coding.

Deciding whether to act on an opportunity. The downward pass identifies opportunities; it does not mandate acting on them. Use this rule: if the Step 4b reuse count for the AREA dimension exceeds 4 and the register budget from Step 2 has more than 32 float-equivalents of headroom, increasing the thread tile is likely worthwhile — revise Step 2's thread-tile entry and recheck the register budget. If either condition is not met, note the opportunity in your derivation record and leave the tile at its current size. For the upward pass, the same logic applies: if moving a REDUCE dimension up one level requires introducing an inter-block reduction kernel or significant synchronization overhead, record the opportunity but assess whether the added complexity is warranted for your problem size.

These observations don't come from profiling. They come directly from reading the index tree structure. The framework makes these opportunities visible by construction, which is the deepest reason the Binary Tree Table exists.

            if (m_out < M && n_out < N)
                C[m_out*N + n_out] = C_acc[tm][tn];
        }
    }
}

// ║  Block-Sparse Flash Attention  (DeepSeek sparse pattern)           ║
// ║  O[q,v] = sparse_softmax_j( Q·K/sqrt(D) ) · V                     ║
// ╚══════════════════════════════════════════════════════════════════════╝

// ── DECLARATIONS [Sub-table 4, row 1] ─────────────────────────────────
// [NUM_WARPS must be a compile-time constant — shared memory arrays require constant sizes]
// [Because warp_partial[NUM_WARPS][T_J] is __shared__, NUM_WARPS must be constexpr, not a runtime int]
// [D_HEAD and T_J are template parameters or #defines; NUM_WARPS is derived at compile time]
static_assert(D_HEAD % 32 == 0, "D_HEAD must be a multiple of 32 for full warps");
constexpr int NUM_WARPS = (D_HEAD + 31) / 32;  // compile-time constant; ceiling division

// [Sub-table 2, smem buffers | Step 2 Block→Shared, validated Step 4a]
__shared__ float K_smem[2][T_J][D_HEAD];
__shared__ float V_smem[2][T_J][D_HEAD];
__shared__ float score_smem[T_J];              // reuse=D_HEAD, highest-reuse buffer
// [Step 2 inter-warp trigger: NUM_WARPS=(D_HEAD+31)/32 → warp_partial required]
__shared__ float warp_partial[NUM_WARPS][T_J]; // NUM_WARPS constexpr above — valid smem size

__global__ void sparse_attention(
    const float* Q, const float* K, const float* V, float* O,
    const int* kv_indices, const int* row_ptr, int S)
{
    // [Symbol legend, grid/thread vars | Step 2 Grid→Global: q→blockIdx.x, k/v→threadIdx.x]
    const int q_idx   = blockIdx.x;
    const int d_idx   = threadIdx.x;
    // [NUM_WARPS is a constexpr defined above — not redefined here; warp/lane derived from d_idx]
    const int warp_id = d_idx / 32;
    const int lane_id = d_idx % 32;

// ── BOUNDS CHECK [Sub-table 4, row 2] ─────────────────────────────────
    // [Sub-table 3, conditional predicate | Step 1b: grid-level leaf → early exit]
    if (q_idx >= S || d_idx >= D_HEAD) return;

// ── INIT [Sub-table 4, row 1 continued] ───────────────────────────────
    // [Sub-table 3, arithmetic AREA | Q: blockIdx.x×D_HEAD + threadIdx.x]
    const float q_reg = Q[q_idx * D_HEAD + d_idx];
    // [Sub-table 2, register state vars | Step 1a: (m,l,o) over j; Algorithm Classification: m=−∞]
    float m_acc = -FLT_MAX;
    float l_acc = 0.f;
    float o_acc = 0.f;

    // [Sub-table 2, tile loop bounds | CSR row_ptr gives sparse tile count]
    const int q_block   = q_idx;
    const int num_tiles = row_ptr[q_block+1] - row_ptr[q_block];

    // [Zero-length guard | if sparse row is empty, write identity and return]
    if (num_tiles == 0) { O[q_idx * D_HEAD + d_idx] = 0.f; return; }

// ── TILE LOOP [Sub-table 4, row 3] ────────────────────────────────────
    // [Sub-table 3, lookup leaf | kv_indices drives iteration — not sequential j]
    for (int tile = 0; tile < num_tiles; ++tile) {
        const int buf = tile & 1;
        // [Sub-table 3, lookup | kv_indices[row_ptr[q]+tile] → physical KV block]
        const int kv_block = kv_indices[row_ptr[q_block] + tile];

// ── LOAD [Sub-table 4, row 4] ─────────────────────────────────────────
        // [Sub-table 3, arithmetic REDUCE + lookup | j_abs from lookup, d_idx direct]
        for (int j = 0; j < T_J; ++j) {
            int j_abs = kv_block * T_J + j;
            // [Sub-table 3, conditional predicate | Step 1b: tile-boundary guard]
            K_smem[buf][j][d_idx] = (j_abs < S) ? K[j_abs*D_HEAD+d_idx] : 0.f;
            V_smem[buf][j][d_idx] = (j_abs < S) ? V[j_abs*D_HEAD+d_idx] : 0.f;
        }
        // [Step 3 FSM, barrier | K_smem, V_smem written; read by all threads in COMPUTE]
        __syncthreads();

// ── COMPUTE [Sub-table 4, row 5] ──────────────────────────────────────
        for (int j = 0; j < T_J; ++j) {
            // [Sub-table 4 row 5 | q_reg reuse=nnz_blocks across tile loop, Step 4b]
            float partial = q_reg * K_smem[buf][j][d_idx];

// ── WARP REDUCE [Sub-table 4, row 6] ──────────────────────────────────
            // [Active mask rule: compute from ballot, never hardcode 0xffffffff]
            // [Step 2 Warp→Primitives: shfl_down | Step 3 cross-check 1: lane_id=k in COMPUTE]
            unsigned mask = __ballot_sync(0xffffffff, true);
            for (int off = 16; off > 0; off >>= 1)
                partial += __shfl_down_sync(mask, partial, off);

// ── BLOCK MERGE write [Sub-table 4, row 7] ────────────────────────────
            // [Step 2 inter-warp trigger: NUM_WARPS=(D_HEAD+31)/32 → warp_partial buffer]
            if (lane_id == 0) warp_partial[warp_id][j] = partial;
        }
        // [Step 3 FSM, barrier | warp_partial written by lane 0; thread 0 reads all]
        __syncthreads();

// ── BLOCK MERGE reduce + score [Sub-table 4, row 7 continued] ─────────
        if (d_idx == 0) {
            for (int j = 0; j < T_J; ++j) {
                float total = 0.f;
                for (int w = 0; w < NUM_WARPS; ++w) total += warp_partial[w][j];
                int j_abs = kv_block * T_J + j;
                score_smem[j] = (j_abs < S) ? total * rsqrtf((float)D_HEAD) : -FLT_MAX;
            }
        }
        // [Step 3 FSM, barrier | score_smem written by thread 0; read by all in SOFTMAX_UPDATE]
        __syncthreads();

// ── NOVEL PHASE: SOFTMAX_UPDATE [Sub-table 4, row 8] ──────────────────
        // [Algorithm Classification: online softmax from output contract]
        // [Step 1a: state vars m_acc, l_acc, o_acc; reduce body = online softmax merge]
        for (int j = 0; j < T_J; ++j) {
            float s_j   = score_smem[j];
            float m_new = fmaxf(m_acc, s_j);
            float e_old = __expf(m_acc - m_new);
            float e_new = __expf(s_j  - m_new);
            l_acc = l_acc * e_old + e_new;
            o_acc = o_acc * e_old + e_new * V_smem[buf][j][d_idx];
            m_acc = m_new;
        }

        // [Step 3 FSM, barrier | smem consumed; next LOAD will overwrite K_smem, V_smem]
        __syncthreads();

    } // end tile loop

// ── NORMALIZE [Sub-table 4, row 10] ───────────────────────────────────
    // [Algorithm Classification: output contract → weights sum to 1 → divide by l_acc]
    o_acc /= l_acc;

// ── STORE [Sub-table 4, row 11] ───────────────────────────────────────
    // [Sub-table 3, arithmetic AREA | symbol legend: q→blockIdx.x×D_HEAD + threadIdx.x]
    O[q_idx * D_HEAD + d_idx] = o_acc;   // no predicate: S blocks cover q exactly
}

// [Sub-table 2, launch config | Step 2 Grid→Global: AREA q → one block per query]
void launch_sparse_attention(float* Q, float* K, float* V, float* O,
                              int* kv_indices, int* row_ptr, int S) {
    dim3 grid(S);        // [symbol legend: q → blockIdx.x, block tile = 1 query per block]
    dim3 block(D_HEAD);  // [symbol legend: k/v → threadIdx.x, D_HEAD threads = 2 warps]
    sparse_attention<<<grid, block>>>(Q, K, V, O, kv_indices, row_ptr, S);
}

Quick reference

Step 1a
Dimension Fate
For each dimension: does it appear in the output? Mark AREA or REDUCE. For every REDUCE dimension, write the reduction body and state variable. Verify the reduction is associative and commutative (parallel character). If it is not — for example, selection (top-K, argmax) or scan (prefix sum) — this framework does not apply; stop and use a dedicated algorithm reference for that operation.
→ dimension fate table, reduction bodies, state variable types, reduction characters
Algorithm
Classification
Which algorithm family implements this reduction? Compute relevant ratios and choose. Write down the output contract — ordering, normalization, stochastic steps — everything the output must satisfy beyond shape. Name every intermediate tensor the chosen family introduces.
→ algorithm family, output contract, any intermediate tensors introduced
Step 1b
Access Patterns
For each dimension of each tensor (including intermediates from the algorithm): what leaf type is the index? Arithmetic, lookup, type-conversion, or conditional? For conditional leaves: is the triggering iterator a grid-level dimension (→ Bounds Check phase, early exit) or a tile iterator (→ predicated load inside the tile loop, never an early exit)?
→ leaf type table; conditional leaf placement (early exit vs predicated load)
Step 2
Binding Table
Record granularity per (dimension, level). Fill four columns: Grid→Global, Block→Shared, Warp→Primitives, Thread→Registers. For Warp→Primitives: after selecting the primitive family, compute NUM_WARPS = (blockDim.x + 31) / 32 (ceiling division) per REDUCE dimension. If NUM_WARPS > 1, add a warp_partial[NUM_WARPS] buffer to Block→Shared and note two extra FSM sync points. Then: (1) smem budget check — sum all Block→Shared buffers including warp_partial; (2) register budget check — sum all Thread→Registers state variables, flag if >64 floats.
→ granularity chain, grid shape, smem layout + size, warp_partial if needed, register size estimate, warp primitive family
Step 4a
Lifetime (block)
Before building the FSM: for every Block→Shared buffer, compute reuse count (product of active iterator sizes absent from the buffer's index). If reuse = 1 and no coalescing justification, remove from Block→Shared and revise Step 2. If large reuse was missed, add to Block→Shared. Redo budget checks. Only proceed to Step 3 when all staging decisions are validated.
→ block-level reuse counts; validated Block→Shared column; any Step 2 revisions resolved before FSM
Step 3
FSM
Read the validated Binding Table for memory boundary crossings to identify phases. Apply the barrier rule: __syncthreads() wherever one thread writes shared memory that another thread reads. For every phase, fill the dim-binding column: "lane_id → [dim]" for warp-primitive phases; "threadIdx → [dim]" for all other phases that touch memory. Apply three cross-checks: (1) lane_id binding matches between Compute and Warp Reduce? (2) REDUCE dimension is parallel in character? (3) threadIdx role is consistent across each shared memory boundary — the buffer is populated and consumed with the same index interpretation? Record which state variables are alive in each phase.
→ ordered phase sequence, barrier locations, dim-binding column (lane_id + thread role), three cross-checks, state variable lifetimes
Opt · Pipeline
(after Step 3)
Only if the Step 4a reuse count is high and profiling shows idle memory cycles: add an Async column to the Binding Table, allocate two shared memory buffers (ping/pong), replace __syncthreads() with cp.async.wait_group(), and add an Init phase before the tile loop.
→ async flag in binding table, double-buffered smem, pipelined FSM — memory and compute overlapped
Step 4b
Lifetime (thread)
After the FSM: for every register-held value, identify which of the thread's owned AREA dimensions are absent from its index. That product is the register-level reuse count. Validate the thread tile shape from Step 2: if tile > 1 but no register value has reuse > 1, reduce tile; if tile = 1 but reuse is large, increase tile. Corrections here do not require redoing the FSM.
→ thread-level reuse counts, validated Thread→Registers tile shape, any Step 2 thread-tile corrections
Step 5
Index Trees
Write every memory address as a tree of typed leaves. Compute max addresses and verify they fit allocated buffers. Upward pass: for every REDUCE leaf, ask what happens if it moves up one hierarchy level (parallelization opportunity). Downward pass: for every AREA leaf at Block or Warp level, ask what happens if a slice moves down into the thread register tile (ILP / register tiling opportunity). Record both sets of opportunities by name before opening an editor.
→ address range verification, upward parallelism opportunities, downward register-tiling opportunities · feeds into: Index expression table max address column
Code
Derivation
Four sub-tables completed in sequence, entirely from the prior steps. (1) Symbol legend: one row per dimension, mapping abstract roles (grid index var, block tile, thread index var, thread tile, reg loop var, memory size) to your kernel's actual variable names — populated from the Step 2 granularity table. (2) Declaration table: launch config, smem buffers, register state vars — read from symbol legend and Steps 2/1a. (3) Index expression table: one row per global memory access, one active column per leaf type — arithmetic AREA uses the cascade formula from the legend, REDUCE uses Flatten→Stride→Unflatten, lookup adds an indirection, conditional adds a predicate. (4) Phase-ordered code skeleton: read FSM rows top to bottom; each row becomes one code block in the sequence shown.
→ symbol legend · declaration table · index expression table with max-address checks · phase-ordered code skeleton
The framework answers "what structure should this kernel have?" — grid shape, memory staging, phase ordering, barrier placement, warp usage. It does not answer "how fast will it be?" Occupancy, register spill counts, instruction throughput, and vectorization width are profiling-time concerns. The framework's scope ends exactly where systematic derivation gives way to hardware-specific tuning.
Scope limitations

I've used this framework on kernels where every REDUCE dimension is associative and commutative (parallel reduction character). The kernels I've worked through with it are: vector add, transpose, shared reduced sum, GEMM, dense attention, paged KV MLA attention, and sparse attention.

It does not cover: scan (prefix sum, exclusive scan, running count — Blelloch up-sweep/down-sweep changes the phase structure to two passes with fundamentally different inter-thread dependencies); selection (top-K, argmax, beam search — requires warp min-heaps, radix select, or bitonic sort, each introducing intermediate tensors the Binding Table is not designed to derive); or any operation whose reduction body is not freely associative and commutative.

If the operation is outside this set, the derivation steps will still run but the phase structure they produce probably won't be right. Catching that at Step 1a saves a lot of rework later.
